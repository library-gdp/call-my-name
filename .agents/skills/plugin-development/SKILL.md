# call-my-name 플러그인 개발

## 작업 경계

- 설치·배포 루트를 `call-my-name/`으로 고정하라.
- 최종 플러그인 사용자에게 필요한 구성요소만 `call-my-name/` 아래에 작성하라.
- 저장소 개발용 `.agents/`, `.codex/`, 루트 `AGENTS.md`와 플러그인 배포물을 구분하라.
- 사용자가 요청한 구성요소와 이를 유효하게 만드는 직접 연관 파일만 변경하라.
- 플러그인 기능을 새로 기획하거나 범용 플러그인 작성 지침을 산출하지 마라.
- 기존 변경을 보존하고 개인정보, 인증 정보, 토큰, 비공개 지원자 자료를 작성하거나 커밋하지 마라.

## 작업 절차

1. 저장소 루트 `AGENTS.md`, `call-my-name/AGENTS.md`가 있으면 해당 파일, `call-my-name/.codex-plugin/plugin.json`, 요청과 관련된 기존 구성요소를 읽어라.
2. 사용자 요청에서 대상 구성요소, 목적, 입력 자료, 기대 동작, 출력 형식과 완료 조건을 추출하라. 결과를 크게 바꾸는 정보가 없을 때만 질문하라.
3. 현재 저장소의 유사 구성요소를 우선 재사용하라. 형식이나 지원 여부가 불확실하면 현재 Codex 플러그인 스키마와 로컬 검증 도구 또는 공식 문서를 확인하고, 기억에 의존해 필드나 경로를 만들지 마라.
4. 다음 기본 위치를 기준으로 대상 파일을 작성하되, 현재 스키마가 다른 위치를 요구하면 검증된 규약을 따르라.
   - Skill: `call-my-name/skills/<skill-name>/`
   - Agent: `call-my-name/agents/<agent-name>.md`
   - Hook: `call-my-name/hooks.json` 및 플러그인 내부의 연관 실행 파일
   - MCP: `call-my-name/.mcp.json`
   - 플러그인 지침: `call-my-name/AGENTS.md`
   - 플러그인 매니페스트: `call-my-name/.codex-plugin/plugin.json`
5. 구성요소의 내용은 사용자가 제공한 요구사항과 프로젝트의 한국어 작성 원칙을 반영하라. 코드 식별자, 경로, 명령어, API 이름은 정확한 원문을 유지하라.
6. 새 파일이 발견되도록 필요한 경우에만 매니페스트를 갱신하라. 실제 구성 파일이 없거나 현재 스키마가 지원하지 않는 매니페스트 필드는 추가하지 마라.
7. 아래 점검을 수행하고 실패 원인을 수정한 뒤 변경 결과를 보고하라.

## 구성요소별 점검

- Skill은 핵심 절차를 `SKILL.md`에 간결하게 두고 상세 자료만 `references/`로 분리하라. `agents/openai.yaml`을 포함하고 스킬 검증기로 구조와 메타데이터를 검사하라.
- Agent는 역할, 작업 범위, 제약, 필요한 입력과 출력 형식을 명확히 정의하고 다른 Agent와 책임을 불필요하게 중복하지 마라.
- Hook은 지원되는 이벤트와 matcher를 사용하고, 명령 경로를 플러그인 루트 기준으로 해석할 수 있게 작성하라. 실패 처리와 부작용을 검토하고 비밀값을 명령에 넣지 마라.
- MCP 설정은 서버 이름, 전송 방식, 실행 명령 또는 URL을 현재 스키마에 맞게 작성하라. 자격 증명은 설정 파일에 직접 넣지 말고 환경 또는 지원되는 인증 흐름으로 주입하라.
- 플러그인 `AGENTS.md`에는 설치된 플러그인이 실행 중 따라야 할 지침만 작성하라. 저장소 개발·커밋·배포 절차는 넣지 마라.
- 매니페스트의 경로는 플러그인 루트에 대한 상대 경로로 유지하고, 참조 대상이 실제로 존재하는지 확인하라.

## 완료 검증

- `call-my-name/`만으로 플러그인 구조가 유효한지 검사하라.
- 플러그인 검증 도구가 있으면 실행하고 JSON과 YAML 구문을 별도로 확인하라.
- `rg --files call-my-name`으로 배포 파일 목록을 검토하여 개발 전용 파일과 민감정보가 섞이지 않았는지 확인하라.
- Skill 제목 구조, 참조 상대 경로, Agent 및 Hook 연관 파일, MCP 서버 참조를 확인하라.
- UTF-8 한국어 표시와 `git diff --check` 결과를 확인하라.
- 실행 가능한 대표 요청이 제공되었으면 새 구성요소가 발견되고 필요한 참조를 로드하는지 검증하라.
- 실행하지 못한 검증과 남은 가정이 있으면 결과 보고에 명시하라.
