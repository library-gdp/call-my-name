# `USER_INSPECTION.md` 출력 계약

아래 세 개의 2단계 제목을 이 순서로 사용한다. 자료에 없는 필드는 만들지 않는다. 빈 섹션은 `- 확인된 정보 없음`으로 표시한다.

## 사용자에 대한 객관적인 사실

가치평가 없이 원자료로 확인되는 사실만 항목별 목록으로 작성한다. 이름·연락처 같은 단일 사실은 한 단계 목록으로, 학력과 경력은 아래의 중첩 구조로 작성한다.

- 이름
- 생년월일 또는 자료에 명시된 나이
- 연락처: 이메일, 휴대전화번호
- 연구경력: 논문·연구내용, 연구실, 담당교수
- 자격증과 어학
- 관련 링크

### 학력 구조

학력은 학교별로 하나의 항목을 만들고 학교의 속성을 하위 목록으로 묶는다. 학교명, 전공명, 학위 등을 같은 수준의 독립 항목으로 나열하지 않는다.

각 학교 항목에는 다음 필수 필드를 정해진 순서로 둔다.

- 학교명
- 학위 종류: 고등학교 졸업, 전문학사, 학사, 석사, 박사 등
- 전공명: 고등학교처럼 전공이 없다고 확인되면 `해당 없음`
- 재학 기간: 가능하면 `YYYY.MM ~ YYYY.MM` 형식
- 학적 상태: 졸업, 재학, 휴학, 수료, 중퇴, 졸업예정 등
- 소재지: 서울, 경기, 해외 국가·도시 등 원자료가 확인하는 범위

자료에서 확인되는 경우에만 다음 선택 필드를 추가한다.

- 부전공 또는 복수전공
- 세부 전공 또는 연구 분야
- 이수학점
- 평점과 평점 만점
- 졸업 논문 또는 학위 논문 제목
- 지도교수
- 주요 수강 과목
- 교내 활동: 동아리, 학생회, 학회 등
- 수상 및 장학금
- 교환학생 또는 해외 연수
- 주요 프로젝트
- 특이사항: 조기졸업, 편입, 전과 등
- 이수과목 및 과목별 평점

```markdown
- 학력
  - 학교명: {학교명} [출처: {파일명#locator 또는 URL}]
    - 학위 종류: {학위 종류} [출처: {파일명#locator 또는 URL}]
    - 전공명: {전공명 또는 해당 없음} [출처: {파일명#locator 또는 URL}]
    - 재학 기간: {YYYY.MM ~ YYYY.MM} [출처: {파일명#locator 또는 URL}]
    - 학적 상태: {학적 상태} [출처: {파일명#locator 또는 URL}]
    - 소재지: {소재지} [출처: {파일명#locator 또는 URL}]
    - {자료에서 확인된 선택 필드}: {값} [출처: {파일명#locator 또는 URL}]
```

학교가 여러 곳이면 `학교명` 항목을 같은 수준에 반복한다. 하나의 학교에서 복수전공, 교내 활동, 프로젝트가 여러 개 확인되더라도 새 학력 항목으로 분리하지 않고 해당 학교 아래에 목록으로 둔다.

### 경력 구조

경력은 고용 관계가 있는 회사별로 하나의 항목을 만들고 회사에서 맡은 역할, 수행한 업무와 프로젝트, 만든 성과를 그 회사 아래에 묶는다. 회사명과 그 회사에서 수행한 프로젝트를 같은 수준의 독립 경력 항목으로 나열하지 않는다. 동일 회사에 재입사했거나 고용 관계가 명확히 분리된 경우에만 별도 항목으로 나눈다.

각 회사 항목에는 다음 필수 필드를 정해진 순서로 둔다.

- 회사명
- 근무 기간: 가능하면 `YYYY.MM ~ YYYY.MM` 또는 `YYYY.MM ~ 현재` 형식
- 고용 형태: 정규직, 계약직, 인턴, 프리랜서 등
- 소속 부서·팀
- 직책·직급: CTO, 팀장, 매니저, 사원 등
- 담당 직무: 백엔드 개발, 제품 기획, 영업 등
- 주요 업무
- 재직 상태: 재직 중, 퇴사

자료에서 확인되는 경우 다음 중요 필드를 회사 항목 아래에 추가한다.

- 주요 프로젝트
  - 프로젝트명
  - 프로젝트 기간
  - 프로젝트 목적
  - 담당 역할
  - 사용 기술
  - 참여 인원
  - 본인의 기여 내용
  - 결과 및 성과
- 기타 성과: 가능하면 전후 수치, 비율, 시간, 비용 등 원자료의 정량 근거를 보존
- 사용 기술 및 도구
  - 프로그래밍 언어
  - 프레임워크
  - 데이터베이스
  - 클라우드·인프라
  - 협업 도구
  - 모니터링 도구

자료에서 확인되는 경우에만 다음 선택 필드를 추가한다.

- 회사 또는 서비스 소개
- 팀 규모
- 관리 인원
- 이직·퇴사 사유
- 수상 및 사내 포상
- 특허·논문·기술 발표
- 해외 근무 여부
- 연봉 및 처우
- 주요 고객사·협력사
- 업무 관련 링크: 서비스, GitHub, 포트폴리오 등

연봉과 퇴사 사유는 개인 컨텍스트에는 보존할 수 있지만 후속 Agent가 일반적인 이력서 내용으로 자동 채택할 정보로 간주하지 않는다.

```markdown
- 경력
  - 회사명: {회사명} [출처: {파일명#locator 또는 URL}]
    - 근무 기간: {YYYY.MM ~ YYYY.MM 또는 현재} [출처: {파일명#locator 또는 URL}]
    - 고용 형태: {고용 형태} [출처: {파일명#locator 또는 URL}]
    - 소속 부서·팀: {소속} [출처: {파일명#locator 또는 URL}]
    - 직책·직급: {직책·직급} [출처: {파일명#locator 또는 URL}]
    - 담당 직무: {담당 직무} [출처: {파일명#locator 또는 URL}]
    - 주요 업무
      - {구체적으로 수행한 업무} [출처: {파일명#locator 또는 URL}]
    - 재직 상태: {재직 중 또는 퇴사} [출처: {파일명#locator 또는 URL}]
    - 주요 프로젝트
      - 프로젝트명: {프로젝트명} [출처: {파일명#locator 또는 URL}]
        - 프로젝트 기간: {기간} [출처: {파일명#locator 또는 URL}]
        - 프로젝트 목적: {목적} [출처: {파일명#locator 또는 URL}]
        - 담당 역할: {역할} [출처: {파일명#locator 또는 URL}]
        - 사용 기술: {기술} [출처: {파일명#locator 또는 URL}]
        - 참여 인원: {인원} [출처: {파일명#locator 또는 URL}]
        - 본인의 기여 내용: {기여} [출처: {파일명#locator 또는 URL}]
        - 결과 및 성과: {성과} [출처: {파일명#locator 또는 URL}]
    - 기타 성과
      - {정량적 변화 또는 결과} [출처: {파일명#locator 또는 URL}]
    - 사용 기술 및 도구
      - {범주}: {기술 또는 도구} [출처: {파일명#locator 또는 URL}]
```

프로젝트가 여러 개이면 `프로젝트명` 항목을 `주요 프로젝트` 아래에서 반복한다. 업무와 성과도 각각 해당 회사 아래에서 반복하며 별도의 경력 항목으로 승격하지 않는다. 경력의 핵심 흐름은 `어디에서 일했는가 → 어떤 역할을 맡았는가 → 무엇을 했는가 → 어떤 결과를 만들었는가`가 드러나게 구성하고, 단순한 업무 나열보다 본인의 기여와 성과를 우선 보존한다.

### 누락·출처·충돌 처리

학력과 경력의 필수 필드는 해당 학교 또는 회사를 식별할 수 있을 때 모두 표시한다. 원자료에서 값을 확인하지 못한 필수 필드는 값을 창작하지 말고 `자료에서 확인되지 않음`으로 기록하며 출처 표시는 붙이지 않는다. 선택 필드는 값이 없으면 필드 자체를 생략한다.

확인된 각 값 바로 뒤에 `[출처: <파일명>#<locator>]` 또는 `[출처: <URL>]`을 붙인다. 서로 충돌하는 사실은 임의로 선택하지 말고 두 값을 모두 출처와 함께 `[상충 정보]`로 표시한다.

## 사용자 역량

원자료와 경험이 직접 뒷받침하는 개별 역량을 목록으로 작성한다. 추상적인 칭찬이나 추측한 숙련도는 쓰지 않는다.

```markdown
- {역량}
  - 근거 경험
    - EXP-001
    - EXP-004
```

근거 경험이 없는 역량은 작성하지 않는다. 같은 역량의 표현이 중복되면 하나로 합친다.

`review-user-profile`에서 사용자가 역량을 직접 확인하거나 정정한 경우에만 예외로 근거 경험이 없는 역량을 유지할 수 있다. 이때 `근거 경험` 블록을 만들지 않는다. `collect-user-context`가 처음 문서를 생성할 때는 이 예외를 적용하지 않는다.

## 사용자의 경험

회사 업무, 교육, 연구, 프로젝트, 아르바이트, 자격·어학 취득, 성적과 학업 수행, 자기소개서·이력서·포트폴리오에서 확인된 사건을 포괄한다. 같은 사건을 여러 문서에서 발견하면 하나의 경험으로 합치고 출처를 모두 남긴다.

```markdown
- EXP-001: {객관적으로 경험한 사실}
  - 출처
    - {파일명#locator 또는 URL}
  - 마주친 문제
    - {원자료에 명시된 문제}
      - 해결 방식 및 과정: {원자료에 명시된 방식과 과정}
  - 느낀점: {원자료에 명시된 느낀점}
```

- 객관적으로 경험한 사실과 출처는 필수다.
- 문제만 제공되고 해결 방식이 없으면 문제만 쓴다.
- 해결 방식, 과정 또는 느낀점이 제공되지 않았으면 해당 줄이나 하위 항목을 생략한다.
- 관련성이 불명확한 경험은 제목 끝에 `[관련성 확인 필요]`를 붙인다.
- 경험 ID는 문서 내에서 유일해야 하며 역량 섹션의 모든 참조 대상이 실제로 존재해야 한다.

## 사용자 검토 출처

`review-user-profile`에서 사용자가 직접 확인하거나 정정한 내용에는 다음 출처 형식을 허용한다.

- 객관적인 사실: 값 바로 뒤에 `[출처: 사용자 확인]`을 붙인다.
- 경험: `출처` 목록에 `사용자 확인`을 추가한다.
- 원출처가 정정 후 내용도 뒷받침하면 함께 유지하고, 정정 내용과 충돌하면 제거한다.

이 출처는 리뷰 과정에서 받은 사용자 피드백에만 사용한다. `collect-user-context`가 원자료를 수집하면서 확인하지 않은 사실에 임의로 붙이지 않는다.
