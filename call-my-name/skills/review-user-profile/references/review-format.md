# 프로필 검토 표시·편집 규칙

## 표 표시

표에는 사용자가 검토할 실제 내용만 담고 원본 Markdown 구조를 그대로 노출하지 않는다.

### 정보

- 기본 사실은 `항목 | 내용` 표로 표시한다.
- `항목`에는 이름, 생년월일, 연락처, 자격증, 어학, 관련 링크 등의 라벨을 쓴다.
- `내용`에는 값과 출처 표시를 함께 둔다.
- 경력, 학력, 연구경력처럼 하나의 항목에 여러 레코드가 있으면 `경력 1`, `학력 1`처럼 이름을 붙여 각각 별도의 `항목 | 내용` 표로 표시한다.

### 역량

- `역량` 한 열로 된 표를 사용한다.
- 사용자에게 보여주는 표에서는 `근거 경험`과 `EXP-*` ID를 제외한다. 파일에 저장된 매핑은 유지한다.

### 경험

- `경험 | 마주친 문제 | 해결 방식 및 과정 | 느낀 점` 네 열로 표시한다.
- `경험` 셀에는 `EXP-*` ID와 객관적으로 경험한 사실을 함께 표시한다.
- 출처는 표에서 제외하되 파일에는 유지한다.
- 한 경험에 문제나 해결 과정이 여러 개면 번호 또는 `<br>` 줄바꿈으로 같은 셀 안에 나열한다.
- 원자료에 없는 선택 필드는 빈 셀로 표시한다.

## 수동 편집 파일

섹션별 파일 이름은 다음과 같다.

| 표시 이름 | 원본 제목 | 작업공간 루트의 파일 |
| --- | --- | --- |
| 정보 | `## 사용자에 대한 객관적인 사실` | `USER_INSPECTION_FACTS.md` |
| 역량 | `## 사용자 역량` | `USER_INSPECTION_CAPABILITIES.md` |
| 경험 | `## 사용자의 경험` | `USER_INSPECTION_EXPERIENCES.md` |

- 새 파일에는 해당 원본 제목과 현재 섹션의 정규 Markdown만 기록한다. 표로 변환한 내용을 기록하지 않는다.
- 파일에는 다른 2단계 제목을 허용하지 않는다.
- `완료`를 받은 뒤 제목과 본문을 파싱하고 `user-inspection-format.md` 계약 및 리뷰 예외를 검사한다.
- 빈 섹션은 `- 확인된 정보 없음`으로만 표현한다.
- 검증 실패 시 `USER_INSPECTION.md`와 수동 파일을 모두 바꾸거나 삭제하지 말고, 사용자가 고쳐야 할 위치와 이유를 알려준 뒤 다시 `완료`를 기다린다.
- 검증 성공 시 `USER_INSPECTION.md`의 대응 섹션만 교체한다. 전체 문서 검증과 원자적 저장이 끝난 다음 수동 파일을 삭제한다.
