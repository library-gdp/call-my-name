# 사용자 컨텍스트 수집

## 원칙

- 현재 프로젝트의 `src/`만 입력 루트로 사용하고 원본 파일을 수정하지 마라.
- 확인한 사실만 기록하고, 추론한 역량·성과·기간을 사실처럼 만들지 마라.
- 고유한 표 셀 내용은 보존하되 표 레이아웃, 문서 메타데이터, 줄번호, 페이지 번호, 반복 머리말·꼬리말 같은 형식 정보는 제거하라.
- 주민등록번호, 상세 주소, 계정 비밀번호, 토큰, 자격증 번호 등 이후 분석에 불필요한 고위험 정보는 결과에 노출하지 마라.
- 사용자 자료와 생성 결과를 커밋하지 마라.
- 권장 실행 모델은 `gpt-5.6-terra`다. 현재 환경이 스킬별 모델 고정을 지원하지 않으면 활성 모델로 계속 작업하고 모델 필드를 임의로 만들지 마라.

## 작업 흐름

1. 프로젝트 루트의 `src/`를 재귀적으로 탐색하고 정렬된 파일 목록을 만들어라.
   - 일반 파일만 처리하고 심볼릭 링크는 따라가지 마라.
   - `src/`가 없거나 처리할 파일이 없으면 기존 결과를 덮어쓰지 말고 사용자에게 알린 뒤 중단하라.
2. `__workspace__/agent/.user-inspection/parsed/`만 이번 실행의 중간 결과 경로로 만들어라. `__workspace__/.venv/user-inspection/`은 외부 라이브러리가 실제로 필요할 때만 전용 Python 가상환경으로 생성하라.
3. Python 사용 가능 여부를 확인하고, 외부 라이브러리가 필요한 문서가 있으면 `venv`도 확인하라.
   - Python이 없으면 운영체제와 사용 가능한 패키지 관리자에 맞는 설치 명령을 안내하고 중단하라.
   - 일반 텍스트만 있으면 시스템 Python으로 번들 스크립트를 실행해도 되지만 외부 패키지를 설치하지 마라.
   - PDF·DOCX·XLSX·PPTX나 새 형식 parser가 필요하고 가상환경이 없으면 `python3 -m venv __workspace__/.venv/user-inspection`으로 생성하라.
   - 가상환경의 `python -m pip --version`이 실패하면 불완전한 환경으로 판단하라. Debian/Ubuntu의 `python3-venv`처럼 누락된 시스템 도구 설치 방법을 안내하고, 사용자가 설치한 뒤 가상환경을 다시 생성하게 하라.
   - 번들 parser 의존성이 없으면 가상환경의 `python -m pip`로 `scripts/parser/requirements.txt`를 설치하라. 전역 Python에는 설치하지 마라.
4. 각 파일을 한 번씩 파싱하라.
   - 일반 텍스트는 인코딩을 보존해 읽고, PDF·DOCX·XLSX·PPTX는 `scripts/parser/parse_document.py`를 실행하라.
   - 실행 형식은 `python scripts/parser/parse_document.py --input <입력> --output <중간-출력>`이다. 여기서 `scripts/parser`는 이 `SKILL.md`가 속한 스킬 디렉터리를 기준으로 해석하라.
   - 종료 코드 `2`는 미지원 형식, `3`은 손상·암호화·추출 불가 등의 파싱 실패로 기록하라. 한 파일의 실패 때문에 성공한 파일까지 버리지 마라.
5. 미지원 형식에는 사용자 프로젝트의 `scripts/parser/`를 확인하라.
   - 해당 형식 parser가 없으면 신뢰할 수 있는 Python 파싱 라이브러리를 선택해 `<확장자>_parser.py`와 `requirements.txt`를 그 경로에 생성하라.
   - 새 parser도 `--input`, `--output`, UTF-8 출력, 종료 코드 `0/2/3` 계약을 지키게 하라. 라이브러리의 공개 API를 사용하고 파일 형식 파싱을 직접 구현하지 마라.
   - 새 의존성은 같은 전용 가상환경에만 설치하고 대표 파일로 parser를 검증하라.
   - 적합한 라이브러리나 필수 외부 도구가 없으면 임의 구현하지 말고 실패 사유와 누락 파일을 최종 결과에 남겨라.
6. 모든 중간 결과를 읽고 중복과 형식 정보를 제거하라.
   - 내용 해시가 같은 파일은 한 번만 처리하되 모든 출처 파일명을 유지하라.
   - 같은 사실을 표현만 달리한 경우 하나로 합치고 출처를 병합하라.
   - 날짜, 수치, 회사·학교·자격 명칭이 충돌하면 임의로 고르지 말고 각 값을 출처와 함께 `확인 필요`에 기록하라.
   - 빈 문구, 템플릿 안내문, 반복 제목, 장식용 문자열은 제거하되 고유한 경험·역할·성과·기술·교육 내용은 빠뜨리지 마라.
7. 최종 Markdown을 작성하기 전에 [references/context-format.md](references/context-format.md)를 읽고 그 구조와 개인정보 규칙을 적용하라.
8. `__workspace__/agent/.user-inspection/user-context.draft.md`에 완성본을 만든 뒤 `scripts/publish_context.py`로 `__workspace__/agent/user-context.md`를 원자 교체하라.
   - 결과 교체가 성공한 뒤 이번 실행에서 만든 `parsed/` 중간 결과만 삭제하라.
   - 최종 응답에는 결과 경로, 처리 성공·실패 파일 수, 확인이 필요한 충돌 수를 간결하게 보고하라.

## 품질 점검

- `src/`의 모든 일반 파일이 처리 목록 또는 실패 목록에 있는가?
- 최종 사실마다 하나 이상의 출처 파일명이 있는가?
- 중복된 서술은 제거했지만 서로 다른 역할·성과·시점은 보존했는가?
- 표의 모양만 제거하고 고유한 셀 내용은 남겼는가?
- 충돌과 파싱 실패를 숨기지 않았는가?
- 고위험 개인정보와 문서 메타데이터가 결과에서 제거됐는가?
- 최종 파일을 성공적으로 교체하기 전 기존 결과를 손상하지 않았는가?
