# 사용자 컨텍스트 Markdown 규격

최종 파일을 다음 제목 순서로 작성하라. 정보가 없는 절은 `확인된 정보 없음`으로 표시하고 제목을 삭제하지 마라.

```markdown
# 사용자 컨텍스트

## 처리 요약
## 기본 정보
## 목표 및 선호
## 경력
## 학력
## 자격 및 교육
## 기술 및 역량
## 프로젝트 및 포트폴리오
## 경험 및 성과
## 확인 필요
## 출처 색인
```

## 기록 규칙

- `처리 요약`에는 확인한 전체 파일 수, 성공 파일, 실패·미지원 파일과 사유를 기록하라.
- 각 사실은 간결한 목록으로 작성하고 끝에 `[출처: 파일명]`을 붙여라. 여러 출처는 쉼표로 병합하라.
- 경력·학력·프로젝트는 확인된 기간 순서로 정렬하되, 불확실한 기간은 억지로 정렬하지 마라.
- 성과 수치는 원문에 있는 경우에만 기록하고 단위와 범위를 보존하라.
- 자기소개서의 가치관이나 협업 방식은 사용자의 명시적 진술 또는 구체적 사례와 함께 기록하라.
- 같은 문장을 여러 절에 반복하지 말고 가장 직접적인 절 한 곳에 두어라. 다른 절에서 필요하면 짧게 참조하라.
- 파일명은 출처 표시를 위해 보존하되 원본 절대 경로, 문서 작성자·수정 시각 같은 파일 메타데이터는 기록하지 마라.

## 개인정보 처리

- 이름, 공개 연락처, 일반 지역, 지원에 필요한 기본 정보는 원문에 있고 목적상 유효할 때만 기록하라.
- 생년월일과 성별은 사용자가 명시적으로 필요하다고 했거나 지원 양식에 필요하다는 근거가 있을 때만 기록하라.
- 주민등록번호, 여권번호, 상세 주소, 금융정보, 계정 비밀값, 인증 토큰, 자격증 번호는 값 자체를 기록하지 마라.
- 제외한 고위험 정보가 있으면 값 없이 `민감정보 제외됨`이라고만 `처리 요약`에 표시하라.

## 충돌과 불확실성

- 상충하는 값을 하나로 합치지 마라. `확인 필요`에 쟁점, 값별 출처, 사용자에게 필요한 확인 사항을 기록하라.
- 파싱 품질이 낮거나 문맥이 끊긴 내용은 확정 사실로 승격하지 말고 `확인 필요`에 원문 요지를 남겨라.
- 모델의 해석은 사실 목록에 섞지 마라. 이후 역량 분석에 유용한 해석이 필요하면 `확인 필요`에 `분석 단계에서 검토`라고 표시하라.
